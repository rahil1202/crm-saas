# CRM Frontend Reusability and Performance Plan

## Objective
Refactor the CRM frontend to reduce page/file size, extract reusable UI and state patterns for tables/forms/buttons/modals, and improve runtime performance without changing core behavior.

## Key Findings
- `frontend/src/features/customers/customers-page.tsx` is the largest hotspot at ~2161 lines and contains list UI, filters, document mode, import/export, multiple modal flows, and CSV download logic.
- `frontend/src/app/dashboard/deals/page.tsx` (~1459 lines), `frontend/src/app/dashboard/leads/page.tsx` (~1353 lines), and `frontend/src/features/partners/partners-page.tsx` (~1397 lines) repeat the same CRUD/list scaffolding.
- Local modal shells are duplicated in multiple large pages, including `customers-page.tsx:702`, `partners-page.tsx:322`, `app/dashboard/leads/page.tsx:391`, `app/dashboard/deals/page.tsx:399`, and `features/partners/partner-profile-page.tsx:188`.
- Column visibility persistence is implemented repeatedly with `window.localStorage` instead of being centralized, even though `frontend/src/components/crm/use-crm-list-state.ts` already provides a reusable storage path.
- CSV export loops and blob download logic are duplicated across customers, leads, deals, partners, and campaigns. Example: `customers-page.tsx:1177-1246`, `campaigns-list-page.tsx:549-599`.
- Profile/detail pages duplicate small display primitives such as `DetailItem` in customer, lead, deal, and partner profile pages.
- Some pages still fetch and process too much on the client. `partners-page.tsx:421-439` loads all partners, then filters/sorts/paginates in memory.
- `customers-page.tsx` repeatedly parses `customer.notes` inside column renderers (`customers-page.tsx:296-343`), which adds avoidable render work.

## Refactor Strategy

### Phase 1: Expand shared CRM primitives
Target files:
- `frontend/src/components/crm/crm-list-primitives.tsx`
- `frontend/src/components/crm/use-crm-list-state.ts`
- new helpers/components under `frontend/src/components/crm/`

Work:
- Extract a shared `CrmModalShell` to replace duplicated modal wrappers.
- Extract a shared `CrmConfirmDialog` for delete/confirm flows.
- Add a reusable `CrmAppliedFiltersBar` / filter-chip row.
- Add reusable loading, empty, and error blocks for list screens if current primitives are insufficient.
- Extend `use-crm-list-state` to cover column storage, common filter removal behavior, and any repeated query-state wiring still living in pages.
- Extract shared `columnVisibilityStorage` helper only if `use-crm-list-state` cannot absorb the remaining usage cleanly.

Expected impact:
- Immediate file-size reduction in every CRM list page.
- Lower maintenance cost by consolidating the highest-frequency UI patterns.

### Phase 2: Standardize the list-page composition
Primary targets:
- `frontend/src/features/customers/customers-page.tsx`
- `frontend/src/app/dashboard/leads/page.tsx`
- `frontend/src/app/dashboard/deals/page.tsx`
- `frontend/src/features/partners/partners-page.tsx`
- `frontend/src/features/campaigns/campaigns-list-page.tsx`

Work:
- Move each page to a common structure built from shared primitives:
  - page header
  - tabs
  - toolbar
  - applied filters row
  - table/document table
  - column settings
  - pagination
  - modal slots
- Normalize naming and responsibilities so pages mostly supply:
  - column definitions
  - filter schema/defaults
  - entity-specific fetch actions
  - entity-specific modal content
- Remove local modal and repeated `localStorage` code from page files.

Expected impact:
- Largest line-count reduction.
- Easier reuse for future CRM modules.

### Phase 3: Split feature-local form modules
Primary targets:
- customer create/edit/quick-update/import flows
- lead create/edit/import flows
- deal create/edit flows
- partner create/edit/delete flows

Work:
- Move large modal bodies into focused feature-local components under each feature folder.
- Reuse existing `components/forms/form-primitives.tsx` where possible instead of adding new abstractions too early.
- Keep state lifting minimal: page owns workflow state, extracted components own presentational form layout.
- Introduce small shared field groups only when reused at least twice with nearly identical shape.

Expected impact:
- Significant file-size reduction without over-abstracting domain-specific fields.

### Phase 4: Extract shared detail/profile primitives
Primary targets:
- `frontend/src/features/customers/customer-profile-page.tsx`
- `frontend/src/features/leads/lead-profile-page.tsx`
- `frontend/src/features/deals/deal-profile-page.tsx`
- `frontend/src/features/partners/partner-profile-page.tsx`

Work:
- Extract shared `DetailItem` and a simple detail grid primitive.
- Extract shared profile section card / side summary card only where markup and spacing are materially the same.
- Keep entity-specific business sections local to avoid premature genericization.

Expected impact:
- Cleaner profile pages and less repeated layout code.

### Phase 5: Performance-focused improvements
Primary targets:
- customers, leads, deals, partners, campaigns list pages

Work:
- Remove repeated parsing in render paths by precomputing derived table data with `useMemo`.
- Lazy-load or defer reference data that is only needed when a modal opens.
- Reduce oversized rerender surfaces by extracting modal content and stable presentational sections.
- Where backend supports it, prefer server-side filtering/sorting/pagination/export over fetch-all client loops.
- Consolidate CSV generation and download logic into a shared helper.
- Use `useDeferredValue` for high-frequency search input only where profiling shows noticeable input lag.
- Consider `next/dynamic` for heavy modal-only content if bundle inspection shows worthwhile savings.

Expected impact:
- Better perceived responsiveness and reduced client work on large datasets.

## Concrete Implementation Order
1. Audit existing list pages and map repeated modal, export, filter-chip, and column-storage patterns to shared replacements.
2. Implement `CrmModalShell` and `CrmConfirmDialog` in shared CRM components.
3. Add shared applied-filter chip bar and any missing list-state helpers.
4. Refactor `customers-page.tsx` first as the reference implementation.
5. Refactor `leads/page.tsx` and `deals/page.tsx` to the same pattern.
6. Refactor `partners-page.tsx`, reusing `use-crm-list-state` instead of page-local query and storage code where feasible.
7. Refactor `campaigns-list-page.tsx` to adopt the same export/storage/shared UI helpers.
8. Extract feature-local form components from the largest page files.
9. Extract shared profile primitives and apply them to customer/partner first, then lead/deal.
10. Run a performance pass and remove remaining fetch-all or repeated parsing bottlenecks that are still material after decomposition.

## Guardrails
- Prefer small correct abstractions over a generic meta-framework.
- Reuse existing primitives in `components/crm`, `components/forms`, and `components/ui` before introducing new layers.
- Do not abstract entity-specific business rules just because the layout is similar.
- Keep shared components slot-based and composable rather than config-heavy.
- Maintain current visual language and route behavior.

## Verification
- Run `npm run lint` in `frontend`.
- Run `npm run build` in `frontend`.
- Smoke-check major CRM flows:
  - customer list and document tab
  - lead list and document tab
  - deal list and document tab
  - partners list and partner profile edit/delete
  - campaigns export flow
- Compare before/after for:
  - line count of largest pages
  - duplicate modal/export/storage logic removed
  - search/filter responsiveness on large lists

## Expected Outcome
- Much smaller list/detail page files.
- Reusable CRM components for tables, modals, filter chips, confirmation flows, and shared detail display.
- Less duplicated code across customers, leads, deals, partners, and campaigns.
- Improved frontend performance from smaller rerender surfaces and reduced client-side repeated work.
